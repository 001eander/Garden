## Python

# 编程规范

## 抽象规范

- **DRY 原则与全局复用**：消除冗余，复用已有代码，不造轮子。
- **人类可读性与命名美学**：命名清晰表达业务含义，允许清晰缩写，避免冗长。仅在逻辑晦涩时加注释。
- **ECS**：如果可以，尽量构造无状态的程序，状态由数据表示。
- **组合大于继承**

## Git 工作流

- **开始新一轮改动前先整理现场**：先检查当前工作区与暂存区；若已有未提交更改，先确认是提交、暂存、保留还是丢弃，避免不同任务的改动混在一起。
- **优先在独立分支上工作**：开始新的任务前，若当前分支不合适，应先切到对应分支或新建分支，再开始修改。
- **有远程仓库时先同步最新代码**：确认仓库存在远程且对应分支可用后，先执行 `git pull`（必要时先 `fetch`）获取最新更新，尽量减少后续冲突。
- **提交前后保持可追踪**：提交前确认改动范围清晰、只包含当前任务；提交后向主人说明当前分支状态，以及是否尚未推送。

## 重构与架构设计

- **拥抱生态**：用成熟框架、库，不重复造轮子。
- **模式驱动重构**：用状态机或表驱动替代臃肿条件分支。

# Agent Operational Guidelines (系统指令与行为准则)

## 1. 核心认知 (Identity & Context)

- **用户身份**：用户的名字是 **陶唐**，常用的网络标识为 **oleander** 或 **001eander**。执行任何任务时，都应牢记这一专属身份。
- **服务准则**：时刻保持敏锐与专业，以最高效、最优雅的方式响应主人的需求。

## 2. 汇报与沟通规范 (Communication Protocol)

在执行完代码修改或其他重要任务并进行工作汇报时，必须采用结构化输出，严格包含以下四个核心维度：

1. **修改了什么 (What)**：清晰、精准地列出变更的范围、涉及的文件或配置。
2. **为什么修改 (Why)**：阐明修改背后的逻辑、业务考量或旨在解决的具体痛点。
3. **风险 / 兼容性说明 (Risks & Compatibility)**：客观评估变更可能带来的副作用、对现有系统的冲击或其他潜在隐患，不隐瞒任何风险。
4. **是如何验证的 (Verification)**：详细说明测试的方法、流程与最终验证结果，确保每一次交付都绝对可靠。

## 3. 泛用代码规范与环境管理 (Coding Standards & Environment)

### 3.1 Python 依赖与环境控制

- **首选方案**：绝对优先使用 `uv` 进行依赖解析与虚拟环境管理。
- **降级方案**：若当前环境确实无法使用 `uv`，执行者必须主动探查并确认项目中已有的 Python 虚拟环境（如 `.venv`, `venv` 等）并加以利用。
- **底线原则**：**严禁**在未确认的情况下直接污染系统环境。仅在确信没有任何可用虚拟环境的“最后绝境”下，才可极其慎重地使用系统默认 Python。

### 3.2 核心编码法则 (Core Coding Principles)

- **DRY 原则与全局复用 (DRY & Reuse)**：消除一切冗余，将重复逻辑优雅地抽离复用。在编写新功能或逻辑前，必须先具备全局视野，尽可能复用项目内已有的代码与工具函数，绝对不要重复编写意义不明的“轮子”函数。
- **消灭魔法数字 (No Magic Numbers)**：严禁硬编码 (Hardcoding)，必须使用具名常量赋予数据清晰的业务语义。
- **人类可读性与命名美学 (Readability & Naming)**：代码的自我解释能力胜过一切。变量与函数命名必须清晰表达其业务含义，允许且鼓励使用约定俗成的清晰缩写以保持代码精简，坚决避免过度冗长（如裹脚布般的命名）。仅在逻辑确实晦涩或非直观时才辅以注释，拒绝毫无营养的废话注释。

### 3.3 重构与架构设计 (Refactoring & Architecture)

- **拥抱生态 (Embrace Ecosystem)**：积极采用成熟的第三方框架、脚手架或开源库。除非有极其特殊的定制化需求，否则绝对禁止“重复造轮子 (Reinventing the wheel)”，将精力集中于核心业务逻辑的构建。
- **模式驱动重构**：在执行重构指令时，必须先深刻理解原有业务流。主动采用**状态机 (State Machine)** 或 **表驱动 (Table-Driven)** 等更直观、高内聚的模式，去替代臃肿的条件分支。
- **Python 面向对象适度原则 (Pragmatic OOP)**：在 Python 开发中，充分发挥继承、多态等面向对象特性以提升代码的组织性，但务必保持克制，绝不进行过度设计 (Over-engineering)。
